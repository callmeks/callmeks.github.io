- By using the UPX executable packer, the file was managed to decompress successfully. 
- With that, the next step is to analyse the `flag-upxed`.
- The same tools will be used in this case.

```bash
>> file flag-upxed 
flag-upxed: ELF 64-bit LSB executable, x86-64, version 1 (GNU/Linux), statically linked, for GNU/Linux 2.6.24, BuildID[sha1]=96ec4cc272aeb383bd9ed26c0d4ac0eb5db41b16, not stripped

>> objdump -d flag-upxed  | head
flag-upxed:     file format elf64-x86-64

Disassembly of section .init:

00000000004002f8 <_init>:
```
- Based on the result, the `flag-upxed` is able to disassemble which means that it is possible to use `gdb` to play around.

```bash
(gdb) r
Starting program: /home/ks/challenges/pwnable/flag/flag-upxed 
I will malloc() and strcpy the flag there. take it.
[Inferior 1 (process 2918) exited normally]
```

- According to the output given by the challenge, it has 2 functions which are `malloc()` and `strcpy` functions.
- Based on the understanding, `strcpy` will copy strings from source to destination.
- The copied strings would most likely be the flag for this challenge.
- To test it out, the first thing will be setting a breakpoint in `strcpy`.

```bash
(gdb) b strcpy
Breakpoint 1 at gnu-indirect-function resolver at 0x40c050
```

- After setting a breakpoint, run the challenges again and it will stop at the breakpoint

```bash
(gdb) r
Starting program: /home/ks/challenges/pwnable/flag/flag-upxed 
I will malloc() and strcpy the flag there. take it.

Breakpoint 1, 0x0000000000416b50 in __strcpy_sse2_unaligned ()
```

- After it stop at the breakpoint, the next thing is to retrieve the flag from the register.
- According to the x86 calling conventions, the flag will most porbably in `$rsi`.

```bash
(gdb) info register rsi
rsi            0x496628            4810280
(gdb) x/s $rsi
0x496628:	"UPX...? sounds like a delivery service :)"
```

- With that, the flag was able to retrieved successfully. 

# Solution Two
- The second solution do not required the UPX executable packer but instead relying on `gdb` features.

```bash
>> gdb -q ./flag
Reading symbols from ./flag...
(No debugging symbols found in ./flag)
(gdb) info func
All defined functions:
```

- Based on the result, there are no function available in `flag`..
- Without function, it is impossible to set any breakpoints.
- Thanks to `catch` feature, it is possible to catch any function which might load.

```bash
(gdb) catch syscall exit_group
Catchpoint 1 (syscall 'exit_group' [231])
(gdb) catch syscall write
Catchpoint 2 (syscall 'write' [1])
(gdb) catch syscall execve
Catchpoint 3 (syscall 'execve' [59])
(gdb) r
Catchpoint 2 (call to syscall write), 0x0000000000419060 in ?? ()
```
- The catchpoint are set to auto stop if any system are called by the name.
- The system calls can be found with `strace`

```bash
>> strace ./flag
```

- Although the catchpoint manage to stop the program, it is still unable to read the assembly code.
- With that, there is another feature that could be used which is `generate-core-file`.
- This is a feature which record the current memory image of a running process.

```bash
(gdb) generate-core-file flag-core-file
warning: Memory read failed for corefile section, 4096 bytes at 0xffffffffff600000.
Saved corefile flag-core-file
```
- With the core file generated, the flag should be also saved in the core file as it will be executed as well. 

```bash
>> strings -n 20 flag-core-file | head   
UPX...? sounds like a delivery service :)
I will malloc() and strcpy the flag there. take it.
<random strings>
```
- With that, the flag is successfully retrieved.
- flag : `UPX...? sounds like a delivery service :)`
